# 2026-04-20 — Phase 1 C5/H1: Graph hash via view struct

## What

`Graph.ComputeHash` now marshals a separate `hashView` projection instead of
mutating the receiver in place. The view excludes both `GraphHash` (which
self-references the output) and `SourceRoot` (an absolute path that varies
per machine).

## Why

Two audit findings collapse into a single small fix:

- **C5** — the previous implementation zeroed `g.GraphHash`, marshaled the
  receiver, and restored via `defer`. Two goroutines calling `ComputeHash`
  concurrently could observe each other's mid-mutation state, producing
  wrong hashes and tripping `-race`. Hashing a view struct never touches
  the receiver.
- **H1** — `SourceRoot` is an absolute filesystem path. Two devs on the
  same commit but different checkout roots produced different
  `ProjectHash` values, so a plan made on one machine always reported
  drift on another. Excluding `SourceRoot` from the hash makes
  `ProjectHash` portable.

## Impact

- `GraphHash` values change for any graph computed by an older binary.
  Plans (`current.plan.json`) are ephemeral and regenerated by every
  `panex plan`, so no on-disk migration is needed; first re-plan after
  upgrade simply produces the new-format hash.
- Cross-machine plan replay (which the audit flagged as broken) now
  works: same source tree → same `ProjectHash` regardless of checkout
  path.

## Quality

- `make fmt && make lint && make test && make build` — all green.
- New tests in `internal/graph/types_test.go`:
  - `TestComputeHash_StableAcrossSourceRoots` (H1 regression)
  - `TestComputeHash_DoesNotMutateReceiver` (C5 regression)
  - `TestComputeHash_ConcurrentSafe` (32 parallel calls under `-race`)
  - `TestComputeHash_ChangesWithContent` (drift detection still works)
- Existing `TestGraphHash_Deterministic` and `TestGraphHash_ChangesOnMutation` unchanged.
