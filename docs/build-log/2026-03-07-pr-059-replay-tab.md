## Problem
- The inspector shell already reserved `#replay`, but the tab was still disabled even after PR57 proved that replaying the latest observed runtime probe payload was useful inside Workbench.
- That left replay discoverability weak and delayed product feedback on whether operators want replay as a history-driven surface rather than only a one-click Workbench action.

## Risk and Mitigation
- Risk: the Replay tab could silently broaden into arbitrary message replay.
- Mitigation: replay history only includes payloads that match the existing `panex.workbench.runtime-probe` contract, and replay still goes through `sendRuntimeMessage(...)`.
- Risk: a dedicated Replay tab could drift from the Workbench quick-action semantics and create inconsistent operator expectations.
- Mitigation: both surfaces use the same observed payload source and the same runtime transport path.

## Teach-back (engineering lessons)
- Design lesson: once a constrained action proves useful in a utility surface, the next product step is often to expose the underlying history explicitly before building more abstraction.
- Testing lesson: replay history stays trustworthy when derivation from timeline state is pure and fully testable without browser fixtures.
- Team workflow lesson: a dedicated tab should only be enabled once its scope boundary is written down clearly enough that reviewers can see what was deferred.

## Next Step
- Decide whether replay should stay constrained to the runtime-probe contract or grow into a broader inspector-side history filter for additional replay-safe message families.
