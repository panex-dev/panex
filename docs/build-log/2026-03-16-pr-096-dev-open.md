## Problem
- After `panex dev` starts, users must manually open `chrome://extensions` and load their unpacked extension. New users often forget this step or mistype the URL.
- There was no quick way to bridge the CLI and browser boundary from a single command.

## Approach (with file+line citations)
- Change 1:
  - Why: add a platform-aware browser launcher behind a testable function variable.
  - Where: `cmd/panex/browser.go:1-24` (new file: `openBrowser`, `defaultOpenBrowserCmd` with `darwin`/`windows`/`linux` branches)
- Change 2:
  - Why: wire a `--open` flag into the `dev` subcommand so users opt in to browser opening.
  - Where: `cmd/panex/main.go:28` (updated usage text)
  - Where: `cmd/panex/main.go:136` (`openFlag` flag registration)
  - Where: `cmd/panex/main.go:180-184` (browser opening call, non-fatal on failure)
- Change 3:
  - Why: cover all flag combinations and failure modes in tests.
  - Where: `cmd/panex/browser_test.go:1-178` (new file: 5 tests + `withStubbedOpenBrowserCmd` helper)

## Risk and Mitigation
- Risk: `openBrowser` calls `exec.Command(...).Start()` which spawns a detached process. If the command name doesn't exist, the error must not stop `panex dev`.
- Mitigation: browser failure is caught and printed as a non-fatal note. Test `TestRunDevOpenFlagBrowserFailureIsNonFatal` verifies this contract.

## Teach-back (engineering lessons)
- Design lesson: browser-opening is inherently platform-specific and side-effectful. The function-variable stub pattern (`var openBrowserCmd = defaultOpenBrowserCmd`) keeps the real logic untouched while giving tests full control over what command would be invoked — no subprocess spawning needed in CI.
